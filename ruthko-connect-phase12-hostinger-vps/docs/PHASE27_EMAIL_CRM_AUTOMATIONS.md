## Environment Variables

Set these in Netlify (Site → Environment variables) or `.env` for local dev:

| Variable | Required | Description |
|---|---|---|
| `EMAIL_PROVIDER` | No | `resend` (default) or `sendgrid` |
| `EMAIL_API_KEY` | Yes | API key for the email provider |
| `RESEND_API_KEY` | Alternative | Existing Resend key (auto-detected if `EMAIL_API_KEY` is absent) |
| `EMAIL_FROM_ADDRESS` | No | From address (default: `Ruthko Connect <onboarding@resend.dev>`) |
| `ADMIN_NOTIFICATION_EMAIL` | No | Where admin alerts go (default: `info@ruthkojobs.com`) |
| `PUBLIC_SITE_URL` | No | Site URL for email links (default: `https://ruthkojobs.com`) |
| `SUPABASE_URL` | Yes | Supabase project URL |
| `SUPABASE_SERVICE_ROLE_KEY` | Yes | Service-role key (Netlify functions only — never in browser JS) |

## Security

- No API keys appear in browser source code.
- Frontend calls `/.netlify/functions/send-email` — the function holds the key server-side.
- `process-crm-automation.js` uses `SUPABASE_SERVICE_ROLE_KEY` server-side only.
- Public users cannot read `crm_email_logs`.
- Admins read email logs based on role (viewer, editor, admin, owner).
- Only owner/admin can delete email logs or resend failed emails.

## Database Tables

### `crm_email_logs`
Tracks every email send attempt with status (`pending` / `sent` / `failed` / `skipped`).

### `crm_automation_rules`
Configurable rules: which submission type triggers which action and after how many hours. Seeded with 6 default rules.

### `crm_automation_tasks`
Tasks generated by automation rules. Status: `pending` / `processing` / `completed` / `failed` / `cancelled`.

## Email Triggers

| Trigger | Email Type |
|---|---|
| Job seeker form submitted | `applicant_confirmation` → applicant + `admin_notification` → admin |
| Employer form submitted | `employer_confirmation` → employer + `admin_notification` → admin |
| Sponsor form submitted | `sponsor_confirmation` → sponsor + `admin_notification` → admin |
| Partner form submitted | `partner_confirmation` → partner + `admin_notification` → admin |
| Volunteer form submitted | `volunteer_confirmation` → volunteer + `admin_notification` → admin |
| Event RSVP submitted | `rsvp_confirmation` → attendee + `admin_notification` → admin |

## Automation Default Rules

| Submission Type | Delay | Priority | Task |
|---|---|---|---|
| job_seeker | 24h | normal | Follow up with job seeker |
| employer | 4h | high | Review employer staffing request |
| sponsor | 24h | normal | Follow up with sponsor prospect |
| partner | 48h | normal | Follow up with partner applicant |
| volunteer | 48h | normal | Review volunteer application |
| event_rsvp | 24h | normal | Process event RSVP |

## Failure Behavior

- If email provider fails: form submission is already saved in Supabase.
- Email log is created with `status: 'failed'`.
- Public user still sees success message.
- Admin sees warning inside CRM Email Logs.
- Owner/admin can resend failed emails from the CRM.

## Phase Integration

Phase 26 `crm-intake-service.js` has stub functions (`notifyAdminOfNewSubmission`, `sendApplicantConfirmation`, etc.). Phase 27's `email-automation-service.js` patches these stubs on `DOMContentLoaded` when both scripts are loaded. The `ruthko:intake-submitted` custom event wires Phase 26 → Phase 27 without file conflicts.
